## Open questions for the user

1. **MV2 recognition vs MV3 enforcement.** Row 2 marks MV2 as `do-not-touch`. Does Spiral intend to *recognise* MV2 manifests (parse them, list the extension in the manager, show a "requires MV2" warning) and never execute them, or should MV2 manifests be rejected at load time? Recognition-only is `S` complexity; full MV2 execution is `L` and requires `webRequest` blocking — which is the entire point of the MV2→MV3 migration.

2. **Extension gallery as a Spiral project capability or external dependency.** Row 45 rates the gallery as `XL` and a `1.0-blocker`. Does Spiral intend to build its own gallery (hosted on spiral.sh, with its own signing infrastructure, its own reviewer queue, its own DSA compliance) or does Spiral intend to reuse the Chromium Web Store / AMO pipelines (load extensions from CWS / AMO into Spiral's extension host)? If the latter, row 45 is `L` (the `externally_connectable` + `management` plumbing is still needed) but not `XL`.

3. **Content script world model.** Row 11 (MAIN world) and row 39 (`userScripts` API) both require a "world" concept per Vortex. If Spiral builds Vortex with a single-world model (all extension scripts share a single isolate), rows 11 and 39 are `L`. If Spiral builds a per-world model (each content script gets its own isolate, matching the Chromium model), rows 11 and 39 are `XL` — the world model is the largest Vortex-side design decision in this chunk.

4. **`userChrome.css` and `userContent.css` as a Spiral deliverable.** Row 52 (userChrome) is `niche` (Firefox-only) and row 51 (userContent) is `widespread`. Does Spiral want to ship both as a `P6` feature, or defer `userChrome.css` as a post-1.0 deliverable? The complexity is `M` in both cases — the engineering cost is in getting the cascade order right (user > author > UA), not in the file-watching.

5. **`proxy` API and `webNavigation` API as priority surfaces.** Rows 37 and 22 are both `mixed` prevalence but `M` complexity. Both are things a privacy-focused browser would want to expose early (proxy for VPN / Tor integration; webNavigation for content blocker development). Should these be marked `P4` instead of `P6`?

6. **`userScripts` as a native surface.** Row 39 (the `userScripts` API) is the native surface that makes Tampermonkey / Violentmonkey work without a full extension host — the engine itself registers user scripts and injects them. If Spiral ships `userScripts` as a first-party API (the browser itself, not an extension, manages the user-script store), this is `M` complexity and a strong privacy signal (no extension intermediaries for the user-script store). Does Spiral want to ship `userScripts` as a native surface?
