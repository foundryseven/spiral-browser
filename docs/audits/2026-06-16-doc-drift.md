## 1. P0 — Must fix before the next packet lands

### P0 #1 — [L] Architectural rule violation (packet 1.6.4, the dep arrow)

- **Code:** `crates/spiral-network/Cargo.toml:16` — `spiral-filter.workspace = true` listed as a regular `[dependencies]` entry.
- **Rule:** `.spiral/rules/architecture.md:16–53` places `spiral-filter` *upstream* of `spiral-network` in the dep DAG. The "down-only" rule (line 55–56) forbids upward arrows.
- **Counterparty:** `crates/spiral-filter/Cargo.toml` does **not** depend on `spiral-network` (the reverse is also not in the graph, so the right answer is **neither**).
- **Description:** In packet 1.6.4 I promoted `spiral-filter` from a dev-dep to a regular dep of `spiral-network` so that `lib.rs` could use `spiral_filter::FilterHook`. This is the *wrong* direction per the canonical dep graph.
- **Fix direction:** Decide the architecture:
  1. **Move the `FilterHook` trait (and `Decision`, `Party`) to `spiral-core`** (the bottom of the dep DAG). `spiral-filter::Filter` implements it. `spiral-network::Client` takes `Box<dyn FilterHook>` from core. *Net effect:* the trait is owned by the lowest crate; the policy engine and the network layer both consume it; no upward dep. **Preferred** — matches the "promote to core if 3+ crates need it" rule (`architecture.md:60`).
  2. **Invert the API:** `spiral-network` exposes a callback-shaped hook (`fn(&str, Party) -> Decision`) and `spiral-filter` adapts. This puts the trait-shaped API in `spiral-network` (still upstream of the policy engine). The callback is a function pointer; no trait impl needed. Works but loses the object-safe `Box<dyn FilterHook>` ergonomics.
  3. **Ratify the dep arrow** (rare; would require an ADR amending the canonical dep graph).
- **Why P0:** every future packet on `spiral-network` will compound the wrong dep. Every future packet on `spiral-filter` that wants to use the network layer will silently *not* see it (because the graph is already wrong; we can't trust it). This must be resolved before 1.6.5 (Gyre) lands.
